Orca Service Mode

Proposal: containerized long-running build orchestrator with WebSocket API, git integration, and nix-based environments.

Overview

Orca today is a CLI binary that runs locally, operates on a local directory, and exits when the build completes. This proposal upgrades it to a persistent service. It pulls git repos, provisions per-project nix environments, runs Claude Code agents, and exposes a REST API with SSE streaming for a live web interface.

Core principle: The engine is unchanged. The existing loop, invoke, eval, supervisor, state, and scope modules work as-is. The service wraps them with remote git, a nix shell, and a WebSocket interface.

Serve Mode

Long-running orca serve process. Accepts build requests via WebSocket. Manages multiple concurrent builds.

Git Integration

Clone remote repos, push branches, create pull requests via GitHub/GitLab API. No local directory required.

Nix Environments

Per-project nix flakes provide toolchains. Shared /nix/store volume for caching. One container image for all projects.

Web Interface

REST API for commands, Server-Sent Events for streaming. Create builds, monitor progress, respond to interventions, view logs — all from the browser.

Deploy + QA Stages

New stage types in the workflow loop. Deploy pushes changes to a test environment; QA agent verifies with browser, SSH, HTTP.

Self-Improvement

Orca can improve its own codebase. Bootstrap pattern: the running version builds and tests the next version.

The orca serve process is the outer shell. It never enters a nix environment itself. When a build is requested, it clones the repo, then spawns nix develop --command orca run ... as a child process. The inner orca run is the existing code path — completely unchanged. It inherits the nix environment, so all claude invocations and eval commands have access to the project's toolchain.

deploy is a command stage (like eval), not a Claude agent. It runs a shell command that pushes changes to a live test environment. qa is a Claude agent with access to Playwright (browser), SSH, and HTTP tools.

Project Typedeploy commandqa verifies via
CLI tool / library none needed Bash (run the tool)
Web app (live reload) none — live reload picks up changes Playwright (browser)
API service ssh server 'cd app && git pull && docker compose restart api' curl / HTTP requests
Full stack (e.g. one-zero) ssh server 'cd app && git pull && docker compose up -d' SSH + Playwright + curl
Orca self-improvement bun run build && deploy-test-instance.sh WS client + curl

Key insight: The complexity of the project under test doesn't affect orca's architecture. It's handled entirely by the spec: what deploy runs and what tools the qa agent has. Orca's engine stays the same.

One Docker image for all projects. Each project declares its toolchain via a nix flake or a packages list. The /nix/store volume persists across builds — first build downloads packages, subsequent builds are instant.

The serve process detects the nix environment type and wraps builds accordingly. Resolution order:

PriorityConditionCommand
1Spec declares nix.flakenix develop {path} --command orca run ...
2Spec declares nix.packagesnix shell nixpkgs#pkg1 nixpkgs#pkg2 --command orca run ...
3flake.nix exists in reponix develop {repoDir} --command orca run ...
4shell.nix exists in reponix-shell {repoDir}/shell.nix --run "orca run ..."
5None of the aboveorca run ... (no nix)
 
# Flake (explicit)
nix:
  flake: true                    # use repo root flake.nix

# Flake (subdirectory)
nix:
  flake: ./backend               # use flake at specific path

# Ad-hoc packages (no flake needed)
nix:
  packages: [nodejs, bun, git]   # nix shell -p

# Disable (even if flake.nix exists)
nix:
  enable: false

Every subprocess (claude, eval commands, deploy scripts) inherits the nix environment automatically. No changes to invoke.ts or eval.ts needed.

REST API + SSE

Commands use standard REST endpoints (curl-testable, auth-middleware friendly). Live streaming uses Server-Sent Events (SSE) — simpler than WebSockets, with automatic reconnect built into the browser's EventSource API.

Forge Module (git/forge.ts)

New module for GitHub/GitLab API interaction. Used by the serve process after a build completes to create PRs. Supports token-based auth via environment variable.

interface ForgeConfig {
  type: "github" | "gitlab";
  token: string;  // from env: GITHUB_TOKEN or GITLAB_TOKEN
}

createPR(config, repo, head, base, title, body): Promise<{ url: string; number: number }>

Self-Improvement (Bootstrap)

%%{init: {'theme': 'dark'}}%% sequenceDiagram participant S as orca serve (v1) participant R as orca run (inside nix) participant C as claude agent participant T as Test Suite S->>S: git clone orca repo S->>R: nix develop --command orca run self-improve.yaml loop Iteration R->>T: bun test (unit tests) T-->>R: FAIL R->>C: analyze failures C-->>R: analysis R->>C: develop fixes C-->>R: code changes end R->>T: bun test T-->>R: PASS R->>T: bun run build (compile binary) T-->>R: OK R-->>S: exit 0 S->>S: git push branch S->>S: create PR Note over S: CI deploys new version

The running orca (v1) builds and tests the next version (v2) using the standard build loop. v2 is just source code being modified by an agent and tested with bun test. The bootstrap is the compiler pattern: the known-good version builds the next version. No recursion, no Docker-in-Docker.

For full integration testing of the serve mode and WS protocol, the workflow adds a deploy + qa stage:

workflow:
  loop: [eval, analyze, develop, deploy, qa]

stages:
  deploy:
    command: "bun run build && ./scripts/deploy-test-instance.sh"
    timeout: 60
  qa:
    toolset: all
    # QA agent hits the test instance's WS API, submits a test build,
    # verifies monitoring works, checks intervention flow

Authentication

Claude Code uses OAuth tokens stored in ~/.claude/.credentials.json. For the containerized service, this file is mounted read-only from the host:

volumes:
  - ~/.claude/.credentials.json:/root/.claude/.credentials.json:ro
ConcernDetailMitigation
Rate limiting Max plan rate limit shared across concurrent builds Agent stages are long-running (minutes). Concurrent builds unlikely to hit limits.
Token refresh Access token has expiry; refresh token may be single-use Long expiry (~1 year). Monitor and handle refresh at the serve layer if needed.
Credential leakage Compromised agent could read the token Mounted read-only. Network-restricted container. Token scoped to Claude Code sessions only.

Container Specification

Dockerfile

FROM nixos/nix:latest

# Enable flakes
RUN echo "experimental-features = nix-command flakes" >> /etc/nix/nix.conf

# Core tools (available outside nix shells)
RUN nix-env -iA nixpkgs.git nixpkgs.openssh nixpkgs.curl nixpkgs.bun

# Claude Code
RUN npm install -g @anthropic-ai/claude-code

# Orca binary
COPY bin/orca /usr/local/bin/orca

VOLUME /nix
VOLUME /data

EXPOSE 7070

CMD ["orca", "serve", "--data-dir", "/data"]

What Does Not Change

The following modules are unchanged. The inner orca run process — which does all the actual orchestration work — is identical to today's CLI binary.

  • Loop engine (engine/loop.ts) — task runner, iteration loop, budget management, stuck detection
  • Invoke (engine/invoke.ts) — Claude Agent SDK, subprocess spawning, scope enforcement
  • Eval (engine/eval.ts) — subprocess runner, parser dispatch
  • Supervisor (engine/supervisor.ts) — escalation detection, retry/stop logic
  • State (state/index.ts) — file-based persistence, BuildState/TaskState types
  • Scope (scope/) — file access enforcement, glob matching
  • Parsers (parsers/) — cargo_test, pytest, json, exit_code
  • Templates (templates/) — variable resolution
  • Notifications (notifications/) — push notifications via command channels
  • Intervention (intervention/) — file-based protocol (serve translates WS ↔ files)
  • Prompts (config/prompts.ts) — 3-tier prompt/schema resolution
  • Config loading (config/loader.ts) — YAML validation
  • Task DAG (config/tasks.ts) — dependency resolution, filtering
